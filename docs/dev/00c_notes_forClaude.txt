Panex Privus

Master software design plan for GitHub package

Official name: Panex Privus

CLI: privy

Mission

Panex Privus is a comparative genomics toolkit for discovering target-group-private genomic signal: alleles, intervals, and regions shared within a target cohort and absent from off-target genomes, with cross-evidence support from VCF, BAM, GFA, and XMFA.

Its primary job is not generic variant processing. Its job is:

Find genomic signal private to the focal cohort, quantify how strong it is, and show whether multiple evidence layers agree.

1. Product definition

Primary use case

Given:
	вЂў	a target group of genomes
	вЂў	an off-target group of genomes
	вЂў	a primary evidence source, usually VCF
	вЂў	optional supporting evidence from BAM, GFA, and XMFA

Panex Privus should:
	1.	identify candidate private alleles and private regions
	2.	merge them into interpretable interval-level findings
	3.	score their support and uncertainty
	4.	compare evidence across modalities
	5.	generate publication-ready summaries and plots

Core commands

The package will ship with four primary commands:
	вЂў	privy scan
	вЂў	privy compare
	вЂў	privy report
	вЂў	privy plot

This is the correct surface area for v1.

вё»

2. Command philosophy

privy scan

Primary discovery engine.

Purpose: detect target-private allele or interval signal, primarily from VCF, with optional BAM/GFA/XMFA support overlays.

This is the main workhorse.

privy compare

Cross-evidence reconciliation engine.

Purpose: compare loci or regions across VCF, BAM, GFA, and XMFA evidence and classify loci as supported, contradicted, source-specific, or uninformative.

This is the intellectual differentiator of the package.

privy report

Interpretation and ranking engine.

Purpose: convert raw scan and compare outputs into ranked summaries, QC tables, and human-readable reports.

privy plot

Focused evidence visualization.

Purpose: explain and diagnose loci with compact, publication-grade figures. Not a genome browser replacement.

3. Core biological logic

Primary discovery model

The central VCF-mode biological rule is:

A locus is interesting when a private allele is present in the target group and absent from the off-target group.

This should be the default and the conceptual center of privy scan.

Supported strictness categories

Missingness should not be buried inside filtering logic. It should be surfaced explicitly as a reported category.

Each locus should carry a column like strictness_class, with values such as:
	вЂў	strict_complete
All targets support the private allele; all off-targets are confidently absent; no missing calls among required samples.
	вЂў	strict_target_missing
Off-target exclusion holds, but one or more target samples are missing or uninformative.
	вЂў	strict_offtarget_missing
Target support holds, but one or more off-target samples are missing or uninformative.
	вЂў	strict_both_missing
Pattern is otherwise consistent, but missingness exists in both target and off-target cohorts.
	вЂў	relaxed_threshold
Passes user-defined support thresholds but not strict-complete logic.
	вЂў	contradicted
Private-allele model fails because at least one off-target supports the allele or target consistency breaks below threshold.

This is the right design because it separates:
	вЂў	biological signal
	вЂў	technical missingness
	вЂў	user policy

That makes the software auditable.

4. Supported evidence types and their roles

4.1 VCF: primary discovery backend

VCF is the main engine in v1.

Supported VCF use cases
	вЂў	multisample VCF
	вЂў	multiple bgzip/tabix indexed VCFs
	вЂў	SNPs
	вЂў	indels
	вЂў	symbolic and interval-based SV representation where possible
	вЂў	phased and unphased genotypes
	вЂў	filtered subsets by QUAL/FILTER/INFO/FORMAT

VCF discovery responsibilities
	вЂў	identify target-private alleles
	вЂў	evaluate target support and off-target exclusion
	вЂў	group nearby discriminant variants into candidate regions
	вЂў	attach missingness class
	вЂў	produce locus-level and region-level outputs

VCF matching modes for v1

v1 should support a clear and limited set:
	вЂў	private_allele вЂ” default
	вЂў	private_genotype вЂ” optional stricter exact genotype logic
	вЂў	private_sv_state вЂ” for symbolic/length-aware SV handling

Do not overcomplicate v1 with haplotype inference beyond what is directly available.

вё»

4.2 BAM: support backend

BAM is support evidence, not a discovery caller.

BAM responsibilities in v1

At candidate loci or intervals, BAM should provide:
	вЂў	read depth
	вЂў	allele counts at known variant positions
	вЂў	allele fraction
	вЂў	reference vs alt support
	вЂў	optional soft-clip / split-read summary near SV-like intervals
	вЂў	coverage sufficiency flags for targets and off-targets

BAM interpretation

BAM should help answer:
	вЂў	is the private allele truly supported in targets?
	вЂў	is off-target absence real or just no coverage?
	вЂў	is there read-level evidence near candidate structural intervals?

BAM non-goals in v1
	вЂў	full de novo variant calling
	вЂў	general-purpose SV discovery
	вЂў	assembly-aware remapping logic

That keeps v1 honest and robust.

вё»

4.3 GFA: graph-context backend

GFA should be treated as a structural context layer, not the master coordinate system.

GFA responsibilities in v1
	вЂў	annotate whether loci fall near graph branches, bubbles, or junctions
	вЂў	report path membership where available
	вЂў	summarize local graph complexity near candidate loci
	вЂў	identify candidate graph evidence supporting structural uniqueness

GFA support model for v1

Use your chosen recommendation:
	вЂў	junction/branch context
	вЂў	path membership
	вЂў	local graph annotation

Not full graph-to-linear universal truth. That way lies chaos.

вё»

4.4 XMFA: optional alignment-evidence backend

XMFA is useful, but secondary.

XMFA responsibilities in v1
	вЂў	alignment-based corroboration of private intervals
	вЂў	support for conserved target-specific alignment states
	вЂў	interval aggregation across discriminant sites
	вЂў	gap-aware regional support

XMFA role

XMFA should be implemented as:
	вЂў	a secondary scan backend if practical
	вЂў	a support layer for compare

But the package architecture should not revolve around XMFA.

5. Internal architecture principle

The package must be format-aware but logic-centered.

Do not let each file format define its own truth. The internal truth model must be:

7. privy scan design

Primary mode

The main entry point should be VCF-first.

Required behaviors

privy scan must:
	1.	ingest cohort definition
	2.	parse VCF in indexed, chunkable fashion
	3.	identify target-private allele patterns
	4.	classify missingness strictness
	5.	merge nearby passing loci into candidate regions
	6.	optionally overlay BAM/GFA/XMFA evidence
	7.	emit machine-friendly outputs plus run metadata

Core scan workflow
	1.	validate inputs
	2.	load cohort and config
	3.	iterate through VCF records by contig/chunk
	4.	evaluate allele patterns
	5.	classify pass/fail and strictness
	6.	emit per-locus hits
	7.	merge loci into regions
	8.	query optional evidence layers around hits/regions
	9.	score hits
	10.	write outputs

вё»

7.1 VCF private allele logic

For each record:
	вЂў	enumerate candidate alternate alleles
	вЂў	compute support in target samples
	вЂў	compute support in off-target samples
	вЂў	classify missingness separately for both groups
	вЂў	determine whether allele qualifies as target-private

Default decision policy

A private allele passes default discovery when:
	вЂў	target support meets threshold
	вЂў	off-target support is zero
	вЂў	required filters pass

Missingness handling

Missingness does not silently fail by default. It is reported as a strictness class and influences score.

This is one of your packageвЂ™s best design choices.

7.2 Region construction

Single variants are not always the biological story. privy scan should support region building.

Region merge strategy

Merge nearby passing loci into regions using configurable parameters:
	вЂў	merge_distance
	вЂў	same_variant_class_only
	вЂў	same_allele_or_state_required
	вЂў	max_gap_with_no_signal

Region-level outputs

Each region should summarize:
	вЂў	number of constituent loci
	вЂў	variant density
	вЂў	target support consistency
	вЂў	off-target exclusion consistency
	вЂў	strictness composition
	вЂў	support from BAM/GFA/XMFA

7.3 Plant pangenome scale requirements

This matters.

The package must assume:
	вЂў	large contigs/chromosomes
	вЂў	high structural diversity
	вЂў	big multisample VCFs
	вЂў	multiple BAMs
	вЂў	possible graph complexity
	вЂў	memory pressure

So privy scan must be designed for:
	вЂў	indexed streaming, not whole-file loading
	вЂў	contig or window chunking
	вЂў	lazy evidence queries
	вЂў	output-first processing
	вЂў	limited in-memory state

That means no naГЇve вЂњload everything into pandasвЂќ architecture in the core engine.

8. privy compare design

This command is where Panex Privus becomes more than a filter.

Purpose

Compare loci or regions across evidence layers and quantify concordance.

Default match logic

You chose the right default:

comparison = interval overlap + evidence compatibility

This should be implemented with source-aware tolerances.

Compatibility dimensions
	вЂў	coordinate overlap
	вЂў	allele/state compatibility
	вЂў	target support agreement
	вЂў	off-target exclusion agreement
	вЂў	evidence sufficiency
	вЂў	contradiction detection

Examples
	вЂў	VCF locus overlaps BAM-supported allele site: supported
	вЂў	VCF SV region overlaps GFA branch context but BAM weak: partially_supported
	вЂў	VCF private allele also present in off-target BAM reads above threshold: contradicted
	вЂў	XMFA interval overlaps region but state ambiguous: uninformative
	вЂў	GFA-only branch with no VCF/BAM/XMFA counterpart: source_specific

Comparison modes

v1 should include:
	вЂў	vcf_vs_bam
	вЂў	vcf_vs_gfa
	вЂў	vcf_vs_xmfa
	вЂў	scan_vs_scan
	вЂў	multi_evidence

Do not expose fifty knobs. Keep it crisp.

11. Scoring system

The scoring system should be transparent and additive.

11.1 Discovery score

Derived from VCF cohort pattern.

Suggested components:
	вЂў	target support fraction
	вЂў	off-target exclusion fraction
	вЂў	private-allele specificity
	вЂў	variant quality contribution
	вЂў	region density contribution if region-level

11.2 Support score

Derived from secondary evidence.

Suggested components:
	вЂў	BAM depth adequacy
	вЂў	BAM allele support
	вЂў	BAM absence confidence in off-targets
	вЂў	GFA junction/branch relevance
	вЂў	path membership support
	вЂў	XMFA corroboration consistency

11.3 Penalty score

Subtract for:
	вЂў	target missingness
	вЂў	off-target missingness
	вЂў	contradictory evidence
	вЂў	low support in key targets
	вЂў	excessive graph complexity if it reduces interpretability
	вЂў	imprecise regional boundaries

11.4 Final score

A visible weighted sum:
final_score = discovery + support - penalty

Weights should live in config and be written into run.json.

That way the method stays explainable and reproducible.

13. CLI spec

Top-level

privy scan
privy compare
privy report
privy plot

Example scan

privy scan \
  --vcf cohort.vcf.gz \
  --targets Benning Harosoy Clark \
  --off-targets Jack Lee Minsoy \
  --mode private_allele \
  --min-target-support 1.0 \
  --max-off-target-support 0.0 \

  --xmfa alignment.xmfa \

  --hits results/hits.tsv \
  --vcf cohort.vcf.gz \
  --bam bam_manifest.tsv \

  --mode multi_evidence \
  --outdir compare/

privy report \
  --hits results/hits.tsv \
  --regions results/regions.tsv \
  --evidence results/evidence.tsv \
  --compare compare/compare.tsv \
  --qc results/qc.tsv \

16. Performance design for plant pangenome scale

This cannot be an afterthought.

Required performance choices
	вЂў	stream indexed VCF/BAM by contig or region
	вЂў	do not keep all records in memory
	вЂў	avoid large DataFrame-first architecture in the core path
	вЂў	support windowed querying of BAM/GFA/XMFA around candidate loci
	вЂў	use manifest-driven sample input for BAMs
	вЂў	parallelize by contig or chunk where safe
	вЂў	log chunk processing statistics

Important implementation note

The package should separate:
	вЂў	discovery pass
	вЂў	support annotation pass
	вЂў	comparison pass
	вЂў	reporting pass

This is more scalable and more debuggable than monolithic everything-at-once logic.
